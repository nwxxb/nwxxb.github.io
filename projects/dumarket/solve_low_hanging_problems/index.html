Solve Low Hanging Problems

Before I deploy this app, I have to fix some “obvious” problems, and before doing that, I set up some scripts to fetch some nice graphs (so we can see the improvement):

  • a simple bash loop that calls docker stat and stores its stdout to a file

  • a ruby script that parses docker stats from point number 1, along with data from the log file (I use lograge to get nicely JSON-formatted lines) and parses it with vega-lite

  • I also initiated a k6 script that covers almost all important endpoints, so we can get the data that we want to graph. Even though the options definition in that file will run the test in a “just visiting” mode instead of “load testing” the feature.

    export const options = {
  scenarios: {
    contact_test: {
      executor: "per-vu-iterations",
      vus: 4,
      iterations: 2,
      maxDuration: "2m",
    },
  },
};

Graphs below show median, p75, and p100 of memory allocation and duration on each endpoint. Each bar has its own color for versions and changes:

  • v0.1: before changes
  • v0.2.1: applying pagination on some index pages
  • v0.2.2: tackle majority of N+1 query
  • v0.2.3: minor change: adding index on one of cart_items foreign key: session_id that only affects POST Users/Sessions#create

So far, changes on v0.2.1 are the ones that brought the most significant performance optimization. The changes on v0.2.2 and v0.2.3 brought uncertain results (some added a small amount of optimization, some did not, and some even made it worse). I feel like there are so many variables that can be the culprit of those slight inconsistencies. However, I postpone digging it out as I feel the context is too vast, and we will profile each of those endpoints later, after I load test it.

To be fair, it’s a bit overkill to generate those graphs since you can manually visit the pages and see the performance through tools like rack-mini-profiler (although it is quite hard to get aggregated data from it). You can also use other tools like lograge, which simplifies your log files and adds some metrics (like memory allocation, request duration, etc) into it, and also don’t forget prosopite gem (might not be as popular as bullet) to detect n+1 queries.
