dumarket is a dummy market app that I use to learn about perf-optimization. codes from this repo initiated with some slow codes.

Solve low-hanging fruit problem

March 5, 2026

Before I deploy this app, I have to fix some “obvious” problems, and before doing that, I set up some scripts to fetch some nice graphs (so we can see the improvement):

  • a simple bash loop that calls docker stat and stores its stdout to a file
  • a ruby script that parses docker stats from point number 1, along with data from the log file (I use lograge to get nicely JSON-formatted lines) and parses it with vega-lite
  • I also initiated a k6 script that covers almost all important endpoints, so we can get the data that we want to graph.

These are two graphs that I generated that show median, p75, and p100 of memory allocation and duration on each endpoint. Each bar has its own color for versions and changes:

  1. v0.1: before changes
  2. v0.2.1: applying pagination on some index pages
  3. v0.2.2: tackle majority of N+1 query
  4. v0.2.3: minor change: adding index on one of cart_items foreign key: session_id that only affects POST Users/Sessions#create

So far, changes on v0.2.1 are the ones that brought the most significant performance optimization. The changes on v0.2.2 and v0.2.3 brought uncertain results (some added a small amount of optimization, some did not, and some even made it worse). I think my dataset is too small for the improvement to be visible. But for now, I am gonna accept it and move to the next step.

To be fair, it’s a bit overkill to generate those graphs since you can manually visit the pages and see the performance through tools like rack-mini-profiler (although it is quite hard to get aggregated data from it). You can also use other tools like lograge, which simplifies your log files and adds some metrics (like memory allocation, request duration, etc) into it, and also don’t forget prosopite gem (might not be as popular as bullet) to detect n+1 queries.
